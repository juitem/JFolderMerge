# 네트워크 드라이브 지원 검토 계획서

**작성일:** 2026-03-25
**브랜치:** feature/network-drive-research

---

## 1. 현황

JFolderMerge는 현재 실행 PC의 **로컬 파일시스템**만 지원한다.
백엔드는 Python 표준 `os`, `pathlib` API를 사용하며, 경로가 존재하면 종류에 관계없이 읽는 구조다.

---

## 2. 검토 배경

- 사용자 환경이 단일 PC에 국한되지 않는 경우 증가
- Obsidian Vault, 작업 파일 등이 NAS나 공유 드라이브에 위치하는 경우
- 현재 `config.json`의 `defaultLeftPath`, `defaultRightPath`가 로컬 절대경로로만 저장됨

---

## 3. 지원 방식 비교

### 방법 A: OS 마운트 네트워크 드라이브 지원 (권장)

**개념:** SMB, NFS, WebDAV 등을 OS 레벨에서 마운트한 후, 마운트 경로를 로컬처럼 사용

| 항목 | 내용 |
|------|------|
| 추가 개발 | 거의 없음 (현재 코드 그대로 동작) |
| 지원 프로토콜 | SMB (`/Volumes/...`), NFS, WebDAV (OS 마운트 시) |
| 안정성 | 높음 |
| 성능 | OS 캐시 활용, 네트워크 속도에 의존 |
| 사용자 부담 | 마운트를 직접 해야 함 |
| 보안 | OS 인증 위임, 앱 수준 추가 처리 없음 |

**필요한 작업:**
- 경로 입력 UI에서 마운트된 드라이브 경로 허용 (현재도 허용됨)
- 연결 끊김 시 에러 메시지 개선 (`os.path.exists` 실패 → 안내 메시지)
- (선택) 경로 브라우저에서 `/Volumes` 포함 표시

---

### 방법 B: 앱 내 네트워크 프로토콜 직접 지원

**개념:** 앱이 직접 SMB, SFTP, WebDAV 등에 접속하여 파일 목록/내용을 가져옴

| 항목 | 내용 |
|------|------|
| 추가 개발 | 대규모 (인증, 연결 관리, 프로토콜별 구현) |
| 지원 프로토콜 | SFTP (`paramiko`), WebDAV (`webdavclient3`), SMB (`smbprotocol`) |
| 안정성 | 구현 품질에 의존, 엣지케이스 많음 |
| 성능 | 직접 제어 가능하나 구현 복잡 |
| 사용자 부담 | 앱 내 인증 정보 입력 필요 |
| 보안 | 자격증명 저장/관리 필요 (보안 리스크) |

**필요한 작업:**
- 프로토콜별 파일시스템 어댑터 추상화 레이어 설계
- 연결 프로파일 UI (호스트, 포트, 인증 방식)
- 인증 정보 안전한 저장 (keychain 연동 등)
- 비동기 파일 읽기 구조로 전환 필요

---

## 4. 결론 및 권장 방향

**방법 A (OS 마운트) 권장**

이유:
1. 현재 아키텍처 변경 최소화
2. 보안/인증은 OS에 위임 → 앱 복잡도 낮음
3. macOS의 경우 Finder에서 SMB/NFS 마운트 후 바로 사용 가능
4. 실제 사용 케이스(NAS, 공유 폴더) 대부분 커버

방법 B는 향후 CLI 없이 원격 서버 간 직접 비교가 필요할 때 재검토.

## 6. 미결 사항

- Windows 환경에서 UNC 경로(`\\server\share`) 지원 여부 확인 필요
- 마운트 끊김 감지 및 재연결 안내 UX 검토
